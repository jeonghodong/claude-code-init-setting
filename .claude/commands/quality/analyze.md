# 특정 원칙 기반 코드 분석

## 실행 지시

**반드시 Task 도구를 사용하여 `code-quality-reviewer` 에이전트를 실행하세요.**

```
Task tool 호출:
- subagent_type: "code-quality-reviewer"
- prompt: "다음 파일을 '$1' 원칙만 집중적으로 분석해주세요: $2"
```

## 분석 원칙: $1
## 분석 대상: $2

## 에이전트가 수행할 작업

`code-quality-reviewer` 에이전트가 지정된 원칙($1)에 대해서만 집중 분석합니다.

## 원칙별 분석 항목

### readability (가독성)
- 한 컴포넌트/함수에 너무 많은 맥락이 섞여있지 않은가?
- 같이 실행되지 않는 코드가 분리되어 있는가?
- 구현 상세가 적절히 추상화되어 있는가?
- 로직 종류에 따라 함수가 적절히 분리되어 있는가?
- 복잡한 조건에 의미 있는 이름이 붙어있는가?
- 매직 넘버가 상수로 정의되어 있는가?
- 코드가 위에서 아래로 자연스럽게 읽히는가?
- 중첩 삼항 연산자가 단순화되어 있는가?

### predictability (예측 가능성)
- 같은 종류의 함수들이 일관된 반환 타입을 가지는가?
- 함수 이름과 실제 동작이 일치하는가?
- 라이브러리와 겹치는 이름을 사용하고 있지 않은가?
- 숨겨진 부수 효과(로깅, 분석 등)가 없는가?
- Discriminated Union이 적절히 사용되고 있는가?

### cohesion (응집도)
- 함께 수정되는 파일들이 같은 디렉토리에 있는가?
- 종류별 분류가 아닌 도메인별 분류가 되어있는가?
- 관련 상수와 로직이 함께 관리되고 있는가?
- 폼의 응집도가 요구사항에 맞게 설계되었는가?

### coupling (결합도)
- 하나의 Hook/함수가 너무 많은 책임을 지고 있지 않은가?
- Props Drilling이 발생하고 있지 않은가?
- 불필요한 공통화로 결합도가 높아지지 않았는가?
- Composition 패턴이나 Context API가 적절히 사용되고 있는가?

## 참조 스킬

- `code-quality-review` 스킬의 해당 원칙 체크리스트를 기반으로 분석

## 출력 형식

1. 해당 원칙($1) 기준 문제점 목록
2. 각 문제에 대한 구체적인 개선 방안
3. Before/After 코드 예시

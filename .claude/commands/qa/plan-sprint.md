월요일 스프린트 계획 미팅에서 사용하는 커맨드입니다. 이번 스프린트의 신규 기능을 분석하고 종합적인 테스트 전략을 수립합니다.

**Sprint Goal or Features**: $1

# Process

## Step 1: 스프린트 정보 수집

다음 정보를 사용자에게 확인하거나 제공받은 정보를 기반으로 진행:

**필수 정보**:
- 스프린트 기간 (예: 2024-11-25 ~ 2024-11-29)
- 신규 기능 목록 및 우선순위
- 인수 조건 (Acceptance Criteria)
- 팀 리소스 가용성 (QA헤드 15h, 클라우드 5h, 디자이너 5h, 개발자 3h)

**선택 정보**:
- 이전 스프린트 이슈 carryover
- 특별한 제약사항 (예: 휴가, 인프라 작업)

만약 정보가 부족하면:
```
다음 정보가 필요합니다:
- [ ] 스프린트 기간
- [ ] 신규 기능 목록 (우선순위 포함)
- [ ] 각 기능의 인수 조건
- [ ] 특별한 제약사항 (있다면)
```

## Step 2: 테스트 범위 산정

**qa-test-planner 에이전트 활용**:
각 기능에 대해:
1. 요구사항 분석
2. 테스트 시나리오 도출 (P0/P1/P2)
3. 테스트 케이스 수 추정
4. 소요 시간 계산

**리소스 제약 확인**:
- 총 필요 시간 vs 팀 가용 시간
- 만약 초과: 우선순위 조정 또는 스프린트 조정 제안

## Step 3: 테스트 계획서 생성

각 기능별로 다음 내용 포함:

```markdown
## [Feature Name]

### 기본 정보
- **Priority**: P0/P1/P2
- **Owner**: [담당자]
- **Acceptance Criteria**: [목록]

### 테스트 범위
- **기능 테스트**: X개 시나리오 (Xh)
- **성능 테스트**: X개 시나리오 (Xh)
- **보안 테스트**: X개 시나리오 (Xh)
- **UI/UX 테스트**: X개 시나리오 (Xh)

### Tier 할당
- **Tier 3 (New)**: 전체 시나리오
- **Tier 1 Migration**: P0/P1 시나리오 (릴리즈 후)

### 리스크
- [High risk area 1]
- [High risk area 2]
```

## Step 4: 주차별 일정 수립

**월요일**:
- 10:00-11:00: 스프린트 계획 미팅
- 11:00-13:00: 테스트 케이스 상세 작성 (qa-test-case-writer 스킬)
- 14:00-15:00: 테스트 환경 준비

**화-목**:
- Tier 3 신규 기능 테스트 실행
- 이슈 발견 시 즉시 리포팅
- 개발자 수정 → 재테스트 사이클

**금요일 오전**:
- 10:00-11:00: 릴리즈 미팅 (Go/No-go)
- 11:00-12:00: 최종 검증

**금요일 오후**:
- 12:00-14:00: 프로덕션 배포
- 14:00-15:00: Smoke test
- 15:00-16:00: 회고 미팅

**버퍼**: 20% (예상치 못한 이슈 대응)

## Step 6: 우선순위 트리아지

만약 리소스가 부족하면:

**Option 1: 기능 축소**
- P2/P3 기능을 다음 스프린트로 연기
- 최소 기능 세트(MVP)만 구현

**Option 2: 테스트 축소**
- P0 시나리오만 집중 테스트
- P1/P2는 smoke 테스트만

**Option 3: 스프린트 연장**
- 릴리즈 일정 조정 제안

사용자에게 옵션 제시 후 결정 요청

# Guidelines

- **현실적으로 계획**: 팀 역량과 시간을 고려하여 실현 가능한 계획 수립
- **우선순위 명확히**: P0/P1/P2를 객관적으로 할당
- **버퍼 포함**: 예상치 못한 이슈를 위해 20% 여유 확보
- **투명하게 소통**: 리소스 부족 시 명확히 표현
- **데이터 기반**: 과거 스프린트 데이터 참고 (가능하면)
- **협업 강조**: QA만의 계획이 아닌 팀 전체 계획
- **유연성 유지**: 스프린트 중간 변경 가능성 고려

# Quality Checks

계획서 완료 전 확인:
- [ ] 모든 신규 기능에 테스트 계획 존재
- [ ] 리소스 할당이 팀 가용 시간 내
- [ ] P0/P1/P2 우선순위 명확히 할당
- [ ] 모든 인수 조건이 테스트로 커버됨
- [ ] 리스크 식별 및 완화 전략 수립
- [ ] 주차별 일정이 현실적
- [ ] Tier 1 마이그레이션 계획 포함
- [ ] Go/No-go 기준 명시
- [ ] Notion 형식으로 정리됨
